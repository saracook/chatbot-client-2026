# Backup System Overview

This document outlines the backup configuration and maintenance scripts for two dedicated backup hosts:

- `carina-archive-home-zfs1`
- `carina-archive-pi-zfs1`

Backups are managed using daily cron jobs (and `anacron`), leveraging `rsync` over `rush` (a wrapper for SSH), and each backup is isolated within the `/zfs/` filesystem using bind mounts.

---

## Daily Cron Jobs

Each host runs a custom script under `/etc/cron.daily/backups` to manage log rotation and trigger backup staging and file transfers.

---

### Host: `carina-archive-home-zfs1`

**Cron Script Path:** `/etc/cron.daily/backups`  
**Log File:** `/var/log/home_backup_cron.log`

**Behavior:**

1. Deletes logs older than 7 days:
   ```
   /z/backup-scripts/home/<UUID>/logs/
   ```
2. Limits concurrent `rsync` processes to 10.
3. Executes:
   - `/z/backup-scripts/stage_home_backups.sh`
   - `/z/backup-scripts/run_this_to_copy_files_dfc1195b-889d-4123-9a99-c223bb9cc830.sh`  
     *(Note: This script is generated by the staging script during each run.)*

---

### Host: `carina-archive-pi-zfs1`

**Cron Script Path:** `/etc/cron.daily/backups`  
**Log File:** `/var/log/pi_backup_cron.log`

**Behavior:**

1. Deletes logs older than 7 days:
   ```
   /z/backup-scripts/8a38c8df-cfcd-4d8c-bee5-ad121dcc50b3/pi/logs/
   ```
2. Limits concurrent `rsync` processes to 10.
3. Executes:
   - `/z/backup-scripts/stage_pi_backups.sh`
   - `/z/backup-scripts/run_this_to_copy_files_pi.sh`  
     *(Note: This script is generated by the staging script during each run.)*

## SSH via Rush

Both backup processes utilize `rush` (Restricted SSH) for secure SSH access and to limit `rsync` connections.

- Make sure `rush` is installed and configured for the backup containers.
- Validate that `acl` and any required dependencies are present in the container OS.

## How the Staging Scripts Work

The `stage_*_backups.sh` scripts do the following:

1. Identify all directories and files to be backed up by searching to a configured depth.
2. Randomize and split the target list into smaller batch files.
3. Generate a script (`run_this_to_copy_files_*.sh`) that executes `rsync` commands using each batch list.
4. Move old log files into a `previous/` subdirectory.

This approach allows parallel, resumable, and tracked syncing across large directory trees.
